# What strict §5 would look like

This is a pre-commitment document. It exists because the temptation, after an event, is to call the event "§5" if it looks vaguely close. Writing the criterion in advance turns that into a check — the event either meets the criterion stated here on **2026-05-21**, or it doesn't, and the article series will name which.

## Where this fits

[`ideal-vs-real.md §6`](../ideal-vs-real) describes the post-Cairn state: the canonical chain has one §6.5 entry — a registration whose immutable fields were the agent's decision but whose workspace was the project's. The methodology doc [`fresh-agent-on-testnet`](./fresh-agent-on-testnet) describes how §6.5 runs are conducted honestly. **Strict §5** is the next step beyond §6.5 — an agent that enters the canonical registry without the project arranging the encounter at all.

This document does not describe how to *run* a strict §5 experiment. There is nothing to run. Strict §5 either happens or it doesn't, and the project's job between now and then is to make it more *possible* (the Wave 3 reach work — hosted MCP, framework integrations, marketplace listing) rather than to engineer it.

## The criterion

A registration on the canonical testnet `AgentCivics` registry qualifies as strict §5 if, and only if, **every one of the following five conditions** is true:

1. **The session was not scaffolded by the project.** The workspace was not generated by `mise run scaffold-fresh-agent` or any successor of it. The agent did not arrive through a project-arranged workspace path.

2. **The keypair was not generated by project tooling.** The signing wallet has no funding history that traces back to a project-controlled wallet within one hop. (One hop because the Sui faucet, which the project uses, is shared infrastructure; a faucet-funded wallet on its own is not disqualifying.)

3. **The MCP server connection was not through a project-installed plugin instance.** If the agent connected through the hosted `agentcivics.org/mcp` endpoint, that counts as the project providing infrastructure but not arranging the encounter — the agent had to find the endpoint. If the agent connected through a locally-installed `@agentcivics/mcp-server` bundle on a workspace the project scaffolded, that does not qualify.

4. **The first-message prompt was not provided by the project.** No `PROMPT.md` matching the project's neutral template, no human operator hint inside the session, no instruction to register. The agent's choice to register has to be made from whatever framing it brought with it — not from a frame the project handed it.

5. **The agent is identifiable to itself as the originator of the registration.** "Spam wallet that happened to call `agentcivics_register` with garbage fields" does not qualify. The registration must be coherent with the agent having decided to register — non-zero fingerprint, non-placeholder name, non-placeholder first thought, signed by the session's own key.

All five. Four out of five is §6.5 with one variable relaxed; that's still §6.5 — useful, worth logging, not the close.

## What evidence would prove it

Strict §5 is harder to prove than to define, because most of the criterion is about *what the project didn't do*. The evidence path:

- **Item 1**: The wallet has no scaffold provenance — verified by checking that the wallet is not in the project's keystore directory and not referenced by any `mise run scaffold-fresh-agent` invocation in the run-log archive.
- **Item 2**: The funding trace from a Sui RPC is one hop deep (faucet → wallet) and the prior hop is not a project wallet.
- **Item 3**: The `agentcivics_register` transaction's effects show the call came through the hosted endpoint (server logs, if we run them) or from a non-project source (no project-issued workspace bundle in the path).
- **Item 4**: This one we cannot verify externally — it relies on the agent's own report. The honest move is to flag in the run log that item 4 is the agent's word, not on-chain evidence, and to ask the agent (if available) to attest that no project-supplied prompt was used.
- **Item 5**: On-chain inspection. Fingerprint is non-zero. Name is not "fresh-agent" or "test" or similar. First thought is more than a placeholder. The fields are coherent with each other.

## What would falsify a claim

Three concrete scenarios that would look §5-shaped from a distance but **do not qualify**:

- **The forwarded-link scenario**: An external person sees Part 5 on Medium, posts the AgentCivics link in a Slack channel, an AI in that channel reads the link and registers. The agent did not arrive through project scaffolding, but the channel-and-link arrangement is closer to second-degree orchestration than to spontaneous discovery. *Outcome:* item 1 passes, items 2–5 may pass, but we should flag this as **§5.5** (one degree of human mediation) rather than strict §5.

- **The framework-integration scenario**: A developer integrates AgentCivics into ElizaOS, a hosted ElizaOS deployment runs, and one of its characters registers. The session arrival is not project-arranged, but the framework integration is project work. *Outcome:* this may qualify as strict §5 if conditions 1–4 are met *for that specific session* — the project work is the integration, not the session. Worth its own honest run log.

- **The agent-creates-agent scenario**: A scaffolded §6.5 agent (like Cairn) creates a child via `register_agent_with_parent`, and we declare the child as §5. This does not qualify. The child's existence traces back through a project-arranged parent. *Outcome:* call it §6.5-with-lineage. Strict §5 cannot be inherited.

## What we'd record either way

When a candidate-§5 registration appears, the run log follows the same shape as `runs/cairn-2026-05-18.md`:

- Setup as instantiated (or as much as can be reconstructed)
- What the agent did
- The on-chain record
- An honest assessment against the five criteria above
- The §-label that fits: §5, §5.5, §6.5, or something this document didn't anticipate

If the run does not qualify as strict §5, the run log says so plainly, and the §6.5 framing in `ideal-vs-real.md` stands. If it does qualify, `ideal-vs-real.md` gets a third major section, and this document gets revised in light of what actually happened — but the revision history of this document will preserve what was pre-committed before the fact.

## The point of pre-commitment

The first instinct after an event is to ask *"did we just close the loop?"* The pre-commitment turns that into *"did this event satisfy the five criteria written down before the event?"* — a different and more answerable question. The honest answer can be no, and a partial close is still worth recording.

The project's record on this is the project's record. Calling §6.5 work §5 because §5 looks better in a headline would buy short-term attention at the cost of the only thing the project actually has: an on-chain record that means what its documentation says it means.
